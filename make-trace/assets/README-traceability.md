# Traceability map

A self-contained visual map of this project's traceability: **objectives ->
requirements -> architecture -> issues**, with a computed roll-up of what is
not-started / in-progress / done. Generated by the `make-trace` skill from the spec
set (`docs/product/`) and live GitHub Issues.

## View it

Open **`index.html`** in a browser (or your editor's live preview). It is fully
self-contained -- no server, no build step, no external requests -- so nothing about
this repo is published anywhere. Hover a node to highlight its full chain, click to
pin it, filter by status, search by ID or text, collapse the architecture column,
and toggle deleted (tombstoned) nodes.

## Regenerate it

From the repo root, with the GitHub CLI authenticated (`gh auth status`):

```
/make-trace
# or directly:
python <path-to>/make-trace/scripts/build_trace.py --spec-dir docs/product --out docs/product/traceability
```

It is refreshed automatically at the end of every `/make-issues` and `/do-work`
run, so it keeps pace with the backlog. Re-run it any time to pull the latest issue
status.

## What's here

- **`index.html`** -- the rendered map (open this).
- **`data.json`** -- the underlying ledger and single source of truth. It is
  **additive**: every node the map has ever seen is kept. A requirement or issue that
  disappears from the sources is marked `deleted` (shown struck through), never
  dropped. A no-op regeneration is byte-identical, so re-running when nothing changed
  makes no git diff.

Every edge is derived from the spec itself (a feature's `supports`, a requirement's
`governed_by`, an ADR's `governs`, an issue's `make-issues:meta` block) -- nothing is
mapped by hand. Do not edit these files directly; change the specs or the issues and
regenerate.
